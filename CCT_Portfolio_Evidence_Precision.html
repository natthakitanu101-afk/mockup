01 / Context

ปัญหาไม่ใช่แบบฟอร์ม
แต่คือกติกาของระบบ

องค์กรต้องประกาศและตรวจสิทธิใหม่เป็นรอบรายปี แม้ข้อมูลหลักของผู้มีสิทธิส่วนใหญ่จะคงเดิม ส่งผลให้งานเดียวกันเกิดซ้ำทั้งฝั่งผู้ยื่น เจ้าหน้าที่สภา และโรงพยาบาล

01

ผู้มีสิทธิ

กรอกข้อมูลเดิมซ้ำ และไม่เห็นสถานะสิทธิของตนเองอย่างชัดเจน

02

เจ้าหน้าที่สภา

ใช้เวลาไปกับการตรวจรายการเดิมแทนที่จะโฟกัสกรณีที่เปลี่ยนจริง

03

โรงพยาบาล

ต้องตรวจสอบข้อมูลข้ามฝ่ายและติดตามความถูกต้องของสถานะล่าสุด

Problem statement — กระบวนการเดิมใช้ “เวลา” เป็นตัวกระตุ้น ทั้งที่สิ่งที่ควรกระตุ้นงานคือ “การเปลี่ยนแปลงของข้อมูล”

02 / Research setup

ฟังคนที่อยู่ใน
กระบวนการจริง

โปรเจกต์เริ่มจาก pain point ขององค์กรผู้ว่าจ้าง แล้วลงรายละเอียดผ่านการสัมภาษณ์และ walkthrough ขั้นตอนทำงานร่วมกับสองหน่วยงานหลัก

18–20

ผู้เข้าร่วมโดยประมาณ จากการสัมภาษณ์ 2 รอบ รอบละ 9–10 คน

Participants
เจ้าหน้าที่สภาคริสตจักร และบุคลากรโรงพยาบาล
Methods
Stakeholder interview · Process walkthrough · Requirement discovery
Research goals
หางานซ้ำ จุดส่งต่อข้อมูล และแยกข้อมูลคงที่ออกจากข้อมูลที่เปลี่ยน

Research limitation

ขอบเขตหลักฐานที่มี

Discovery รอบนี้เน้น stakeholder ฝั่งปฏิบัติการ หากยังไม่มีการสัมภาษณ์ผู้มีสิทธิปลายทางโดยตรง สมมติฐานด้านความเข้าใจและความสะดวกของผู้ใช้ควรถูกตรวจสอบเพิ่มใน usability test

หลักการนำเสนอ: ไม่อ้างว่าได้วิจัยผู้ใช้ทุกกลุ่ม และไม่สร้าง quote หรือ metric ที่ไม่มีบันทึกจริง

03 / Key insight

Time-based →
Event-based

เปลี่ยน model ของงาน: บันทึกข้อมูลหลักครั้งแรกและเปิดงานตรวจเฉพาะเมื่อเกิด event ที่มีผลต่อสิทธิ

Before

Annual cycle

1ประกาศรอบใหม่
2กรอกข้อมูลเดิม
3ตรวจทั้งหมดอีกครั้ง
4แจ้งผลและเก็บเอกสาร
→

After

Change events

1ตั้งค่าข้อมูลครั้งแรก
2เห็นสิทธิปัจจุบัน
3เกิด event ที่เกี่ยวข้อง
4ตรวจเฉพาะส่วนที่เปลี่ยน

STRUCTURAL EFFECT

เปลี่ยนสิ่งที่กำหนดปริมาณงาน

Annual model
เคสที่ต้องตรวจ = ผู้มีสิทธิทั้งหมดในทุกรอบปี

Event-based model
เคสที่ต้องตรวจ = ผู้สมัครใหม่ + ผู้ที่ข้อมูลเปลี่ยนจริง

ปริมาณงานที่ลดลงจริงขึ้นอยู่กับอัตราการเปลี่ยนแปลงข้อมูลในแต่ละปี ซึ่งยังไม่มีข้อมูล Production เพียงพอสำหรับคำนวณเป็นเปอร์เซ็นต์
เปลี่ยนตำแหน่งย้ายที่อยู่เปลี่ยนโรงพยาบาลเปลี่ยนประเภทสิทธิข้อมูลครอบครัวเปลี่ยน

05 / Design decisions

ทุกหน้าจอมีเหตุผล
จากงานจริง

ไม่แก้เพียงหน้าตา แต่ลดสิ่งที่ต้องอ่าน ต้องกรอก และต้องตรวจในแต่ละจุดของ workflow

ข้อมูลส่วนใหญ่คงเดิม→แสดงเฉพาะส่วนที่เกี่ยวกับ event→ลดการกรอกซ้ำ
เจ้าหน้าที่ต้องหาสิ่งที่เปลี่ยน→Change diff + event queue→ตรวจได้ตรงจุด
สถานะสิทธิไม่ชัดเจน→Rights overview + timeline→สร้างความมั่นใจและตรวจสอบย้อนกลับ

07 / Edge cases & trust

ระบบสิทธิต้อง
อธิบายตัวเองได้

ความไว้วางใจไม่ได้มาจากความสวย แต่จากสถานะที่ชัด สิทธิการเข้าถึง และประวัติที่ตรวจสอบย้อนกลับได้

ข้อมูลไม่ตรงกัน

แสดงแหล่งที่มา วันที่อัปเดตล่าสุด และช่องทางส่งหลักฐานเพิ่ม

Event ถูกปฏิเสธ

บอกเหตุผลและ next action โดยไม่ปล่อยให้ผู้ใช้เจอ dead end

สิทธิย้อนหลัง

แยก effective date ออกจากวันที่อนุมัติและรักษาประวัติทุก revision

Privacy & permission

จำกัดข้อมูลตาม role พร้อม audit log สำหรับการดูและแก้ข้อมูลสำคัญ

ยังไม่มีการสร้างผลทดสอบสมมติ แผนด้านล่างระบุสิ่งที่จะตรวจเพื่อยืนยันว่า workflow ใหม่เข้าใจได้และลดภาระจริง

ผู้มีสิทธิดูสิทธิและอัปเดต event

09 / Expected impact

ผลลัพธ์ที่คาดหวัง
ไม่ใช่ตัวเลขที่แต่งขึ้น

ก่อนมีข้อมูล production ผลกระทบทั้งหมดถูกระบุเป็น hypothesis สำหรับวัดผลในขั้นถัดไป

ลดงานซ้ำ

ให้เจ้าหน้าที่ตรวจเฉพาะรายการที่มีการเปลี่ยนแปลง

เห็นสิทธิชัดขึ้น

ให้ผู้ใช้เข้าถึงสถานะล่าสุดและเข้าใจว่าต้องทำอะไรต่อ

ตรวจสอบย้อนหลัง

สร้างหลักฐานของทุก event การอนุมัติ และวันที่มีผล

Reflection

Research เปลี่ยนโจทย์จาก “ทำฟอร์มให้ดีขึ้น” เป็น “หยุดสร้างงานที่ไม่จำเป็น”

Case study นี้นำเสนอเฉพาะข้อมูลที่จำเป็นต่อการอธิบายกระบวนการออกแบบ และไม่แสดงข้อมูลส่วนบุคคลของผู้เกี่ยวข้อง
