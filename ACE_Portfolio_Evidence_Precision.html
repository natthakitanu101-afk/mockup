02 / Discovery inputs

ข้อมูลจริงจากธุรกิจ
แต่ไม่ใช่ direct user research

การตัดสินใจเริ่มจาก requirement และ pain point ที่ทีม Sales ได้รับจากลูกค้า แล้วนำมาวิเคราะห์ร่วมกับระบบเดิมและผลิตภัณฑ์ในตลาด

01

Sales insights

รวบรวม requirement, feature request และ pain point ที่ลูกค้าโรงเรียนแจ้งผ่านทีม Sales

02

Competitive analysis

เปรียบเทียบ feature coverage, workflow pattern และช่องว่างของคู่แข่งที่เกี่ยวข้อง

03

Heuristic review

ตรวจระบบเดิมด้าน feedback, navigation, touch target, consistency และ error prevention

LIMITATIONข้อมูลผ่านตัวกลางและอาจมี sales bias จึงถือหลายข้อเป็น design hypothesis ไม่อ้างว่าเป็นการสัมภาษณ์ครู นักเรียน หรือผู้ปกครองโดยตรง และต้อง validate กับผู้ใช้ปลายทางในขั้นถัดไป

03 / Problem hypotheses

สามบทบาท
เชื่อมด้วยข้อมูลชุดเดียว

ข้อสังเกตในส่วนนี้สังเคราะห์จากข้อมูลลูกค้าผ่านทีม Sales, product review และ competitive analysis จึงเป็น working hypotheses ที่ยังต้องตรวจสอบกับผู้ใช้ปลายทางโดยตรง

EVIDENCE LEVELBusiness-informed hypotheses · ยังไม่ใช่ผลจาก direct user research
TEACHER

ครู

ต้องจัดการงานประจำวันและเผยแพร่ข้อมูลให้หลายฝ่าย

  • มีแนวโน้มทำงานซ้ำใน Excel และหลายระบบ
  • การเช็กชื่อ ออกเกรด และรายงานอาจมีหลายขั้นตอน
  • อาจต้องตอบคำถามเกี่ยวกับข้อมูลเดิมซ้ำ
STUDENT

นักเรียน

ต้องเข้าถึงผลการเรียน เอกสาร และสถานะคำขอด้วยตนเอง

  • อาจเข้าถึงผลและเอกสารได้ไม่ทันที
  • บางขั้นตอนยังต้องพึ่งครูหรือเจ้าหน้าที่
  • สถานะคำขออาจไม่ชัดเจน
PARENT

ผู้ปกครอง

ต้องติดตามข้อมูลสำคัญโดยไม่เพิ่มภาระการสื่อสารให้ครู

  • ข้อมูลการเรียนและการเงินอาจอยู่คนละจุด
  • การแจ้งเตือนอาจไม่ทันต่อการตัดสินใจ
  • บางกรณีต้องสอบถามครูเพื่อยืนยันสถานะ
WORKING HYPOTHESIS — workflow และข้อมูลสำคัญหลายส่วนอาจยังอยู่นอกระบบร่วม ทำให้เกิดการส่งต่อข้อมูลและงานซ้ำระหว่างบทบาท

ลด context switching

เพิ่ม self-service
